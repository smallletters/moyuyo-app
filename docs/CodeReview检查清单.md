# Code Review 检查清单

## 前端检查项

### TypeScript 类型
- [ ] 是否存在 `any` 类型滥用？应优先使用 `unknown` 或具体类型
- [ ] 函数返回值类型是否明确声明
- [ ] 泛型是否合理使用，避免过度抽象
- [ ] `interface` 与 `type` 的使用是否遵循项目约定
- [ ] 枚举值是否使用 `const enum` 或联合类型替代普通枚举（减少运行时开销）

### 组件 Props
- [ ] Props 是否包含完整的类型定义（使用 `defineProps` 或 `PropType`）
- [ ] 非必传 Props 是否设置了合理的默认值（使用 `withDefaults`）
- [ ] Props 命名是否遵循驼峰式，模板中使用短横线分隔
- [ ] 是否为布尔类型 Props 提供了明确的 `default: false`
- [ ] 是否避免了过深的 Props 传递，应考虑 `provide/inject` 或状态管理

### Vue 模板性能
- [ ] `v-for` 是否绑定了唯一的 `key`（避免使用 `index` 作为 key）
- [ ] `v-if` 与 `v-show` 的选择是否合理（高频切换用 `v-show`，低频用 `v-if`）
- [ ] 是否避免在模板中调用复杂方法或计算属性
- [ ] 大数据列表是否使用了虚拟滚动
- [ ] 组件是否合理拆分，避免单个组件过大

### 样式规范
- [ ] 颜色值是否使用 CSS 变量而非硬编码（如 `var(--color-primary)` 而非 `#1890ff`）
- [ ] 字体大小/间距等是否使用设计系统 token
- [ ] 样式作用域是否限定（`scoped` 或 CSS Modules）
- [ ] 是否存在无用的样式代码

### 网络请求
- [ ] 是否通过封装的 `request.js`/`http.ts` 发送请求
- [ ] 错误处理是否到位（统一错误提示、401 跳转登录、网络异常处理）
- [ ] 是否合理使用了请求/响应拦截器
- [ ] 请求是否在组件销毁时取消（`AbortController` 或 `onUnmounted`）
- [ ] API 地址是否使用环境变量，而非硬编码

### 国际化
- [ ] 文案是否通过 `t()` 或 `$t()` 引用，而非硬编码字符串
- [ ] 动态拼接的文本是否使用了 `:args` 参数化方式
- [ ] 日期、数字格式化是否使用了国际化工具

### 其他
- [ ] 是否有未使用的 import
- [ ] 是否有 console.log 残留（调试代码应移除）
- [ ] 图片资源是否有懒加载处理
- [ ] 表单是否有必要的防抖或节流处理

## 后端检查项

### API 参数校验
- [ ] 请求体是否使用了 `@Valid` 或 `@Validated` 注解
- [ ] 字段级别是否使用了 `@NotBlank`、`@NotNull`、`@Size`、`@Pattern` 等校验注解
- [ ] 路径变量和查询参数是否有 `@PathVariable`、`@RequestParam` 校验
- [ ] 自定义校验逻辑是否使用 `@Constraint` 注解封装

### SQL 与数据库
- [ ] 是否存在 N+1 查询问题（循环中查询数据库）
- [ ] 是否使用了 MyBatis-Plus `QueryWrapper` / `LambdaQueryWrapper`，而非手写 SQL（除非必要）
- [ ] 复杂查询是否使用了分页（`Page` 对象），避免全量查询
- [ ] 批量操作是否使用了批量方法（`saveBatch`、`updateBatchById`）
- [ ] SQL 中是否使用了参数占位符 `#{}` 而非字符串拼接

### 事务管理
- [ ] `@Transactional` 注解范围是否正确（仅在需要事务的方法上使用）
- [ ] 是否合理设置了 `rollbackFor`（默认只回滚 `RuntimeException`）
- [ ] 事务方法中是否包含了 RPC 调用或长时间操作（导致事务过长）
- [ ] 同一类内的事务方法调用是否失效（自调用绕过代理）

### 日志安全
- [ ] 是否使用 `@Slf4j` 或 Logger 记录日志
- [ ] 敏感信息（密码、手机号、身份证、Token）是否被打印
- [ ] 日志级别是否合理（开发用 `debug`，生产用 `info`/`warn`/`error`）
- [ ] 异常日志是否包含了足够的上下文信息

### 异常处理
- [ ] 业务异常是否被全局异常处理器（`@RestControllerAdvice`）捕获
- [ ] 是否使用了自定义业务异常类（如 `BusinessException`）而非直接抛 `RuntimeException`
- [ ] 异常信息是否对用户友好，避免暴露内部实现细节
- [ ] 是否处理了 `Feign` / RPC 调用的异常降级
- [ ] 文件上传等资源操作是否有 finally 释放资源的逻辑

### 单元测试
- [ ] 核心业务逻辑是否有单元测试覆盖
- [ ] 测试是否包含了正常路径和异常路径
- [ ] 是否使用了 Mock 框架隔离外部依赖
- [ ] 测试方法命名是否清晰（`testXxx_shouldDoXxx_whenXxx`）

### 安全
- [ ] 用户输入是否有 XSS 过滤处理
- [ ] SQL 是否存在注入风险（确认使用参数化查询）
- [ ] 接口是否有权限控制（`@PreAuthorize` 或拦截器）
- [ ] 敏感接口是否有频率限制或防重放机制
- [ ] 文件上传是否有类型和大小校验

### API 文档
- [ ] Controller 是否有 `@Tag` 注解描述模块
- [ ] 接口是否有 `@Operation` 注解说明功能
- [ ] 请求参数和响应字段是否有 `@Schema` 注解描述
- [ ] 响应结果是否统一封装（`R<T>` 或 `Result<T>`）
