## 1. 项目概述

### 1.1 项目名称
**MOYUYO ATELIER**（魔芋友 / MoyuYo 宠物生活）

### 1.2 项目定位
面向欧美中高端养宠家庭的 Modern Luxury 宠物生活方式电商平台，提供宠物洗护、服饰、出行装备、家居用品等品类的一站式购物体验。

### 1.3 项目目标

| 阶段 | 时间 | 目标 |
| :--- | :--- | :--- |
| MVP 上线 | T+21 周 | 完成核心交易闭环，App Store + Google Play 上线 |
| 增长期 | 上线后 4~9 个月 | 月活 ≥ 30 万，月 GMV 突破 $500K USD |
| 成熟期 | 上线后 10~18 个月 | 月 GMV 突破 $2M USD |

### 1.4 版本范围（MVP）

| 模块 | 功能范围 |
| :--- | :--- |
| **用户系统** | 注册/登录（Email + Apple/Google/Facebook 社交登录）/ 2FA / 用户档案 / 地址管理 / GDPR 合规 |
| **商品系统** | SPU/SKU / 搜索 / 分类 / 详情页 / 评价 / 收藏 |
| **交易链路** | 购物车 / 下单 / 支付（Stripe/Apple Pay/Google Pay）/ 订单管理 / 退款 |
| **物流履约** | 订单发货 / 物流追踪 / WooCommerce 同步 / 亚马逊 MCF 集成 |
| **营销活动** | 满减 / 秒杀 / 优惠券 / 积分 / 签到 / 邀请 |
| **会员体系** | 5 等级 / 成长值 / 积分抵现 / 专属权益 |
| **Pet Hub** | 3D 互动场景 / 护理记录 / 健康日历 / 成就系统 |
| **订阅服务** | Subscribe & Save / 周期配送 |
| **后台管理** | 商品 / 订单 / 会员 / 营销 / CMS |
| **合规** | GDPR / CCPA / COPPA / PCI-DSS |

### 4.1 代码缺陷修复与优化（T1.1）

**目标**：解决所有已知的 P0/P1 级别缺陷，确保核心交易链路零阻塞。

**检查范围**：

| 检查项 | 验收标准 | 责任人 |
| :--- | :--- | :--- |
| 后端编译无错误 | `mvn clean install` 通过 | 后端开发 |
| 前端编译无错误 | `npm run build` 通过 | 前端开发 |
| 代码规范检查通过 | Checkstyle + ESLint 无 ERROR | 全团队 |
| API 响应格式统一 | 所有接口遵循 `Result<T>` 格式 | 后端开发 |
| 异常场景处理完善 | 服务端异常不暴露堆栈，返回友好错误信息 | 后端开发 |
| 空数据/边界值处理 | 列表空数据展示空状态，分页边界值正确 | 前端开发 |
| iOS/Android 双端原生插件稳定 | 插件无崩溃、无内存泄漏 | 原生开发 |

### 4.2 支付流程全链路联调（T1.4）

**测试场景**：

| 场景 | 支付方式 | 预期结果 |
| :--- | :--- | :--- |
| 正常支付 | Stripe 信用卡 | 支付成功，订单状态更新，Webhook 回调正常 |
| 正常支付 | Apple Pay | 支付成功，钱包卡片生成 |

## 7. 灰度发布策略

### 7.1 灰度流程

```
阶段 1：内部验证（1 天）
  └── 内部团队 + 种子用户（≤ 50 人）
  └── 监控：Crash 率 = 0，核心接口 P99 < 500ms

阶段 2：1% 灰度（1 天）
  └── 随机 1% 用户
  └── 监控：Crash 率 < 0.5%，异常率 < 1%

阶段 3：5% 灰度（1 天）
  └── 随机 5% 用户
  └── 监控：Crash 率 < 0.3%，异常率 < 0.5%

阶段 4：20% 灰度（1 天）
  └── 随机 20% 用户
  └── 监控：Crash 率 < 0.1%，异常率 < 0.2%

阶段 5：全量开放（100%）
  └── 所有用户
  └── 持续 72h 监控值守
```

### 7.2 灰度暂停条件

任一条件触发立即暂停灰度并回滚：

| 指标 | 阈值 | 动作 |
| :--- | :--- | :--- |
| App Crash 率 | > 0.5% | 暂停灰度，回滚至上一版本 |
| 下单接口错误率 | > 2% | 暂停灰度，回滚 |
| 支付接口错误率 | > 1% | 暂停灰度，回滚 |
| 接口 P99 延迟 | > 2s | 暂停灰度，排查性能问题 |
| 支付成功率 | < 98% | 暂停灰度，排查支付通道 |
| 用户负面反馈 | 集中投诉 > 5 条 | 评估后决定是否暂停 |

### 7.3 回滚方案

| 回滚场景 | 回滚方式 | 回滚时间 | 影响范围 |
| :--- | :--- | :---: | :--- |
| App 客户端严重 Bug | 应用商店下架 + 引导用户不升级 | 即时 | 新版本用户 |
| 后端服务异常 | K8s 回滚 Deployment 到上一版本 | < 5 分钟 | 所有用户 |
| 数据库 Migration 异常 | 执行回滚 SQL | < 10 分钟 | 影响可逆 |
| 支付通道故障 | 切换到备用支付通道 | < 2 分钟 | 部分用户 |
| 第三方服务不可用 | 降级策略启动 | 即时 | 部分功能受限 |

### 8.2 应急预案

#### 8.2.1 支付通道故障

```
触发条件：Stripe/PayPal API 连续错误率 > 5%

响应流程：
1. 自动切换到备用支付通道（Stripe → PayPal / PayPal → Stripe）
2. 通知运营团队通过 Zendesk 发布公告
3. 开发团队排查故障原因
4. 修复后灰度恢复支付通道
5. 72h 内输出故障报告

通信模板：
  - App 内：显示 "Payment service temporarily unavailable, please try again later."
  - 客服：话术统一 "我们正在修复支付服务，预计 X 小时内恢复"
```

#### 8.2.2 数据库故障

```
触发条件：MySQL 主库不可用 / 复制延迟 > 30s

响应流程：
1. 自动切换到只读模式（用户可浏览不可下单）
2. 触发 PagerDuty 告警通知 DBA
3. DBA 排查故障原因（主库宕机/复制延迟/慢查询）
4. 执行故障恢复（主从切换/慢查询 Kill/索引优化）
5. 确认恢复后解除只读模式
6. 72h 内输出故障报告
```

#### 8.2.3 安全事件

```
触发条件：检测到数据泄露/入侵/大规模刷单

响应流程：
1. 立即隔离受影响系统
2. 触发 P0 告警通知安全负责人
3. 收集证据（日志/访问记录/快照）
4. 评估影响范围（数据泄露量/受影响用户数）
5. GDPR 72h 内通知监管机构
6. 通知受影响用户
7. 修复漏洞 + 改进安全策略
8. 72h 内输出安全事件报告
```

| MySQL 数据库 | 每日全量 + 每 5min binlog | 30 天全量 / 7 天 binlog | AWS S3 |
| 用户文件（头像/评价图） | 实时同步 | 永久 | AWS S3（跨区域复制） |
| 配置文件 | Git 版本管理 | 永久 | GitHub |
| 日志文件 | 每日归档 | 30 天热 / 180 天冷 | AWS S3 + Glacier |
| 订单数据 | 实时 + 每日对账 | 7 年 | AWS S3 Glacier Deep Archive |
